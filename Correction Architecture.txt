Architecture révisée — Système de gestion scolaire Django

1. Règles de nommage adoptées
Avant tout, voici les conventions appliquées uniformément dans toute l'architecture :
Langue : français partout pour les noms métier (modèles, vues, URLs). L'anglais est réservé aux termes purement techniques imposés par Django ou les standards REST (token, api, JWT, slug).
Modèles : PascalCase en français — AnneeScolaire, FraisScolaire, EleveInscription.
Vues et URLs : snake_case en français — liste_eleves, creer_paiement, detail_classe.
Champs de modèles : snake_case en français sans exception — est_approuve, est_actif, date_creation.
Noms de pages : toujours sous la forme [Action] — [Objet] — "Créer un élève", "Modifier un paiement", "Liste des classes".

2. Structure des applications
L'architecture passe de 7 à 8 applications avec des responsabilités clairement séparées.
core
Rôle : configuration centrale et entités partagées par toutes les autres apps. C'est ici que vivent les concepts transversaux.
Modèles : AnneeScolaire, Cycle, PeriodeEvaluation, NiveauScolaire, ConfigurationEtablissement.
Middleware : ActiveAnneeScolaireMiddleware (timeout 30 minutes avec avertissement UI), JournalAuditMiddleware.
Context processors : contexte_annee_scolaire, contexte_notifications.
authentification
Rôle : tout ce qui concerne les utilisateurs, les accès et la sécurité. Anciennement accounts.
Modèles : Utilisateur, ProfilUtilisateur, Role, Permission, PermissionUtilisateur, GroupeUtilisateur, QuotaRole, Notification, Message, JournalAudit, TentativeConnexion, CodeReinitialisation, DemandeApprobation.
Champs unifiés sur Utilisateur : nom_utilisateur, courriel, role, telephone, adresse, est_approuve, est_actif, date_creation. Les champs métier comme salaire_base, date_embauche et matiere sont supprimés ici et déplacés dans les apps concernées.
scolarite
Rôle : tout ce qui concerne les élèves, leur vie dans l'établissement et leur suivi. Anciennement eleves.
Modèles : Eleve, ParentTuteur, DossierMedical, DocumentEleve, EleveInscription, SanctionRecompense, ConduiteEleve, ConfigurationConduite, CloturePeriode, ClotureNotes.
enseignement
Rôle : la gestion académique et pédagogique. Anciennement academics.
Modèles : Classe, Matiere, CoefficientMatiere, Salle, Professeur, Attribution, Evaluation, Examen, Epreuve, FicheNote, SeanceHoraire, ContrainteHoraire.
Note importante : le modèle Professeur est ici et est lié à Utilisateur par une relation OneToOneField. Il n'existe plus dans finances. Fin du doublon.
presences
Rôle : suivi quotidien de la présence des élèves. Inchangé dans la responsabilité, renommé dans les détails.
Modèles : AppelEleve, Presence, SeanceCours, JustificationAbsence.
finances
Rôle : la gestion financière de l'établissement, sans les données RH qui appartiennent à ressources_humaines.
Modèles : FraisScolaire, Paiement, Facture, LigneFacture, ChargeFixe, ChargeOperationnelle, BourseRemise, EleveBourse, RappelPaiement, CategorieDepense, OperationCaisse, RapportFinancier.
Note : AnneeScolaire et Cycle sont importés depuis core et non redéfinis ici.
ressources_humaines
Rôle : gestion du personnel et des salaires. C'est une nouvelle app qui centralise ce qui était éparpillé entre accounts (salaire_base, date_embauche) et finances (Personnel, Salaire).
Modèles : MembrePersonnel, FichePoste, Salaire, FichePaie, ContratEmploye.
MembrePersonnel a une relation OneToOneField vers Utilisateur, exactement comme Professeur. Toute personne de l'établissement est d'abord un Utilisateur, puis spécialisée par son profil métier.
rapports
Rôle : génération asynchrone de documents. Les vues ne génèrent plus de PDF directement — elles créent des tâches de génération.
Modèles : Bulletin, RapportAcademique, RapportFinancier, TacheGeneration.

3. Unification des profils utilisateurs
C'est la correction la plus importante. Le projet original avait trois représentations d'un même employé. Voici le modèle unifié :
Utilisateur (authentification)
│
├── ProfilProfesseur (enseignement)    ← OneToOneField → Utilisateur
├── MembrePersonnel (ressources_humaines) ← OneToOneField → Utilisateur
└── CompteParent (scolarite)          ← ForeignKey → Eleve
Un professeur est un Utilisateur avec role='professeur' ET un ProfilProfesseur associé. On ne duplique jamais nom, prenom, telephone, courriel — ces données ne vivent que dans Utilisateur.

9. Corrections techniques prioritaires
Base de données : PostgreSQL obligatoire dès le départ. SQLite est interdit en environnement multi-utilisateurs. La documentation ne doit plus présenter SQLite comme option par défaut.
Génération de PDF : toutes les générations de PDF (bulletins, fiches de paie, factures, reçus) passent par une tâche asynchrone Celery. La vue HTTP crée une TacheGeneration, retourne immédiatement une réponse 202, et le client interroge un endpoint /rapports/taches/<uuid>/statut/ pour connaître l'avancement.
Timeout de session : porté de 5 minutes à 30 minutes. Un avertissement est affiché dans l'interface 5 minutes avant l'expiration, avec un bouton "Prolonger la session" qui effectue un appel silencieux pour réinitialiser le compteur.
Champs JSON remplacés : horaires dans Attribution devient un modèle SeanceHoraire avec des champs jour, heure_debut, heure_fin. Les champs actions et approbateurs dans Permission deviennent des relations ManyToManyField vers ActionPermise et Utilisateur.
API REST : séparation des sérialiseurs en lecture et écriture. Chaque ressource a un LectureSerializer (tous les champs, imbrications) et un EcritureSerializer (champs validés uniquement, pas de données dérivées).

Ce que cette architecture corrige par rapport à l'originale : zéro doublon de modèles, une seule convention de nommage sans mélange de langues, des responsabilités d'apps clairement délimitées, des concepts partagés centralisés dans core, une gestion du personnel unifiée dans ressources_humaines, et des décisions techniques (PostgreSQL, Celery, timeout) adaptées à un ERP en production réelle.
